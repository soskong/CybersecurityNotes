```c
define unlink(P, BK, FD) {   
    // P = chunk1 = 0x130f010
    FD = P->fd;   // FD = 0x6022e8 ,FD为P的前驱
    BK = P->bk;   // BK = 0x6022f0 ,BK为P的后继
    if (__builtin_expect (FD->bk != P || BK->fd != P, 0))	//判断P的前驱的bk是否为P，以及判断P的后继的fd是否为P

//BK->fd = *(0x6022f0+0x10) = 0x130f010	== P

    BK->fd = FD;   // &(0x6022f0 + 0x10) = &0x602300 = 0x6022e8
```

##### fastbin double free

该利用的产生原因是，fastbin在释放时next chunk不清空prev_in_use字段，导致可以多次释放

且由于fastbin中的chunk仅通过fd连接，导致fd容易被劫持

wiki上的例子

首先申请两个堆块，并释放掉

```
free(chunk2);
```

现在是fastbin->chunk2->chunk1

再free掉chunk1

```

```

会出现fastbin->chunk1->chunk2->chunk1

此时再分配一次

```
malloc(0x10)
```

变成fastbin->chunk2->chunk1->chunk2

由于刚刚的分配得到了chunk1，修改chunk1指向tar_addr

```

```

变成fastbin->chunk2->chunk1->tar_addr

tar_chunk处的0x8偏移处即size字段，要符合该fastbin大小才行，一般利用0x7f，分配chunk大小0x60-0x68

##### house of spirit

该技术的核心在于在目标位置处伪造 fastbin chunk，并将其释放，从而达到分配指定地址的 chunk 的目的

要想构造 fastbin fake chunk，并且将其释放时，可以将其放入到对应的 fastbin 链表中，需要绕过一些必要的检测，即

* fake chunk 的 ISMMAP 位不能为 1，因为 free 时，如果是 mmap 的 chunk，会单独处理。
* fake chunk 地址需要对齐， MALLOC_ALIGN_MASK
* fake chunk 的 size 大小需要满足对应的 fastbin 的需求，同时也得对齐。
* fake chunk 的 next chunk 的大小不能小于 `2 * SIZE_SZ`，同时也不能大于 `av->system_mem` 。
* fake chunk 对应的 fastbin 链表头部不能是该 fake chunk，即不能构成 double free 的情况。

wiki上后续对fastbin attack的讲解都是一个思想，劫持fastbin的fd指针，实现任意地址chunk分配

#### unsortedbin attack

究其利用原因时因为被取出的victim chunk的bk指针篡改，而寻找后一个chunk是依赖此指针的，后一个chunk就被劫持为我们设计好的fakechunk

因为victim要unlink，所以前一个即fakechunk的fd要指向unsortedbin的fd，即fakechunk+0x10处的内容被改为unsortedbin的fd地址

unsorted bin可以将内存任一地址修改为较大的值

然而在高版本libc中添加了这样一句检测

```c
if (__glibc_unlikely (bck->fd != victim) || __glibc_unlikely (victim->fd != unsorted_chunks (av)))
	malloc_printerr ("malloc(): unsorted double linked list corrupted");
```

fakechunk的fd被检测，使得unsortedbin attack失效

unsortedbin attack并且会使unsortedbin的bk指针指向fakechunk，可能会破会unsortedbin结构

##### 总结

由于largebin没有针对于 `(fwd->bk_nextsize == p || bck ->fd_nextsize==p)`的检查，导致伪造了一个fakechunk在fwd指向的chunk之后

inseted chunk插入时，使得fakechunk+0x20即fd_nextsize指向了inseted chunk，而inseted chunk的bk_nextsize指向了fakechunk

由于没有针对此的检查，后续也没有针对于插入bin时（fd和bk的操作）的检查，导致了进一步针对fwd的bk的利用

这部操作完成后，链表结构（跳表和bin）都为fakechunk->insterted chunk -> fwdchunk

但是若fakechunk的bk或bk_nextsize不完整伪造的话，链表结构会被破环，当下一次遍历largebin时程序崩溃

以上利用的前置条件：

* 可以完全修改一个chunk的内容
* 插入的chunk要紧跟在fakechunk后，largebin中chunk从大到小，即 `inserted chunk > fakechunk && inserted chunk < fakechunk->bk_nextsize`

利用：

可以将一个地址的值修改为堆块地址，例如劫持_IO_list_all 便于伪造 _IO_FILE 结构体进行 FSOP

以上内容均为libc2.23版本下的，当libc版本来到2.35之后，新增了两条检查：

1. 当该chunk大于此bin中最小chunk时，对跳表结构检查

   ```
   if (__glibc_unlikely (fwd->bk_nextsize->fd_nextsize != fwd))
   	malloc_printerr ("malloc(): largebin double linked list corrupted (nextsize)");
   ```
2. 添加了对于入bin时的检查

   ```
   if (bck->fd != fwd)
   	malloc_printerr ("malloc(): largebin double linked list corrupted (bk)")
   ```

当受到这两条限制时，针对于largebin的利用

对应的 bin 中只有一个 free_chunk，此时 fd_nextsize,bk_nextsize 都指向自己
